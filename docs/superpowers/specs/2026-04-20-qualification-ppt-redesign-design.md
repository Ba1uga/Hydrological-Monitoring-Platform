# 资格赛 PPT 重设计说明

## 目标

为“中国高校计算机大赛-网络技术挑战赛（2026）”普通 A 赛项的资格赛，重做一版更适合网络评审的项目汇报 PPT，使其在匿名提交前提下，更有效地呈现作品的创意、技术关联与应用价值。

本次重设计的直接目标不是“做一套炫酷展示稿”，而是“做一套评委能快速读懂、快速判断作品价值的资格赛说明材料”。

## 设计依据

### 1. 竞赛阶段与评审形式

根据竞赛规程：

- 普通 A 赛项资格赛的申报形式是作品说明材料
- 资格赛评审方式为网络评审
- 资格赛评分结构为：
  - 创意 `40%`
  - 技术 `30%`
  - 应用 `30%`

因此，本次 PPT 的首要任务是帮助评委在短时间内判断：

1. 作品是否属于网络技术挑战赛普通 A 的合理选题
2. 作品是否具备明确创新点
3. 作品是否完成了较可信的系统实现
4. 作品是否具备实际应用价值

### 2. 匿名规则约束

根据资格赛匿名规则说明：

- 所有提交评审的作品资料都不得出现学校、团队或队员身份相关信息
- 包括文字、图片、视频材料
- 若出现可识别身份信息，可能按零分处理

因此，本次 PPT 设计必须天然适配匿名提交，不应依赖成员介绍、学校信息、校徽、校名或带有地理位置指向的识别元素来建立信任感。

### 3. 项目上下文

项目本身是“基于云边协同的智慧水文监测与预警可视化平台”，面向水文监测、防汛抗旱、水位分析、预测预警和工程展示等场景，仓库中已具备：

- Spring Boot 后端统一服务框架
- MySQL / Redis / WebSocket 等技术支撑
- 首页总览、防汛抗旱、水位监测、预测预警、水利工程等多个专题页面

这决定了 PPT 既要保留“国家级监测平台”的专业气质，又不能做成“系统截图拼贴板”。

## 核心设计结论

### 主视觉方向

采用“水文行业辨识度的网络技术评审简报风”。

这一定义具体包含：

- 封面使用深色基调建立专业感与权威感
- 正文多数页面转为浅色背景，提升网络评审阅读舒适度
- 视觉纹理使用抽象化的水脉、节点、拓扑连线、等高线元素
- 避免发光 HUD、霓虹蓝、控制台式重装饰与系统界面直搬
- 强调“说明材料”属性，而不是“发布会”或“路演舞台”属性

### 视觉气质

目标气质为：

- 专业
- 稳重
- 可信
- 有技术感
- 有行业属性

不追求：

- 夸张炫技
- 科幻大屏感
- 花哨动画感
- 互联网产品发布会感

## 视觉系统规范

### 色彩策略

采用“深色封面 + 浅色正文”的结构化色彩体系。

建议角色如下：

- 主深色：墨青黑 / 深石板青
  - 用于封面、章节过渡、重点视觉基底
- 主浅色：浅砂纸白 / 温和米灰
  - 用于正文背景，支撑长时间阅读
- 辅助色：青灰绿 / 石板灰
  - 用于卡片、结构线、信息分区
- 强调色：低饱和琥珀棕
  - 只用于强调创新点、结论、评分关联信息

明确禁止：

- 泛科技蓝大面积使用
- 紫蓝渐变
- 荧光青高亮
- 纯黑与纯白硬碰撞

### 字体与层级

字体风格应服务于“评审材料”而非“海报设计”。

建议层级：

- 封面标题：强识别、厚重、稳
- 正文大标题：高对比、短句式
- 正文说明：左对齐、短段落、低噪音
- 截图说明：小号、功能性文本、统一样式

文本规则：

- 每页只保留一个主结论
- 正文尽量避免长段落
- 关键句使用粗体或强调色，不用大面积彩色文字
- 不使用装饰性下划线、发光描边或渐变字

### 图形母题

整套 PPT 的视觉母题固定为：

- 水脉流线
- 拓扑节点
- 等高线 / 河网抽象线
- 稳定、规整、低噪音的框架化结构

这些元素只能作为背景辅助，不得抢占正文与截图的阅读中心。

## 版式策略

### 页面总原则

整套 PPT 要体现“评委扫描效率优先”。

每页应满足：

1. 3 秒内能看到本页核心主题
2. 8 秒内能看懂本页结论
3. 15 秒内能判断本页与评分项的关系

因此，版式上采用：

- 大标题 + 短副标题
- 结论先行
- 卡片化信息组织
- 截图证据与说明文本分离
- 大留白，不做满屏铺满

### 页面结构模式

整套 PPT 优先使用以下模式：

1. 封面页
   - 深色底
   - 标题 + 简短定位语
   - 抽象水文 / 网络纹理

2. 结论型正文页
   - 左侧主结论 / 解释
   - 右侧结构卡片 / 图示

3. 架构说明页
   - 中间为网络技术关联图或架构图
   - 周边为精炼说明，不堆砌术语

4. 截图证据页
   - 统一边框
   - 统一标题条
   - 每张截图只证明一类能力

5. 总结页
   - 价值、场景、后续规划
   - 收束答辩逻辑

不采用：

- 全文同一种标题 + 子弹列表模板反复复制
- 一整页铺满 3-4 张截图无说明
- 把系统 UI 当 PPT 版式直接使用

## 资格赛与展示版双版本策略

### 资格赛匿名提交版

这是主版本，也是本次优先产出的版本。

特点：

- 严格匿名
- 面向网络评审
- 优先说明创意、技术、应用
- 页数控制在 `8-9` 页左右

拟定页序：

1. 封面
2. 项目背景与问题定义
3. 选题与网络技术赛项的对应关系
4. 项目创新点
5. 技术架构
6. 核心功能与系统完成度
7. 应用价值与落地场景
8. 后续规划与总结
9. 截图证据页（可并入第 6 页或单列）

### 展示 / 汇报版

这是辅助版本，用于非匿名场景。

在匿名版基础上，可加入：

- 成员与分工页
- 更完整的项目汇报信息
- 需要公开展示时使用的身份信息

该版本不作为资格赛匿名提交材料。

## 内容表达策略

### 1. 背景页

重点不是泛泛讲行业背景，而是快速指出：

- 当前水文监测场景的实际问题
- 信息分散、协同不足、实时研判能力不足
- 为什么需要统一监测与预警可视化平台

### 2. 赛项对应关系页

这一页非常关键，用于帮助评委确认作品与普通 A 赛项高度相关。

应明确说明作品与以下方向的关联：

- 网络化数据组织与统一服务发布
- 云计算 / 边缘协同的系统构想
- WebSocket 等实时通信能力
- 监测数据、预警状态、站点联动的网络化支撑
- 网络技术与行业应用的结合

这一页应是资格赛版本中的高权重页面。

### 3. 创新点页

创新点不宜空泛，应避免“系统全面、界面美观”这类弱表述。

应聚焦：

- 多业务主题一体化整合
- 网络化监测 + 实时推送闭环
- 水文行业场景下的多专题可视化表达
- 后续云边协同与智能分析能力的扩展潜力

### 4. 技术架构页

该页必须避免做成普通三层架构模板图。

应体现：

- 数据来源与业务对象
- 数据支撑与服务组织
- 实时通信机制
- 多专题页面如何消费这些能力
- 与“网络技术挑战赛”主题的对应关系

### 5. 功能与截图页

截图页的定位是“系统完成度证据页”。

规则：

- 截图不是背景，而是证据
- 每张截图只证明一个重点
- 截图要有统一说明文案
- 不堆满，不拼贴，不杂乱

推荐证明点：

- 首页总览：全局态势感知能力
- 防汛抗旱：地图联动、实时预警与资源调度
- 水位监测：动态监测与异常发现
- 预测预警：分析与辅助判断能力

## 匿名合规规范

资格赛匿名版必须遵守：

- 不出现学校名称
- 不出现成员姓名
- 不出现团队名
- 不出现校徽、校训或校园标识
- 不出现能直接识别学校或地理位置的截图元素
- 若截图内有可能泄露身份的元素，应裁切或打码

特别说明：

原始需求中的“成员、分工”内容，只能放入展示版，不进入资格赛匿名版。

## 质量标准

最终 PPT 需要满足以下标准：

### 内容标准

- 能在有限页数内清楚说明背景、技术、创新与价值
- 能让评委快速判断其属于普通 A 赛项合理选题
- 能体现作品已经形成可展示系统雏形，而非停留在概念层

### 视觉标准

- 有统一且稳定的视觉系统
- 不是普通模板感
- 不是大屏系统复刻感
- 适合长时间网络阅读

### 合规标准

- 资格赛版本满足匿名要求
- 内容无身份泄露风险
- 页面结构适合提交评审

## 不做的事情

本次设计不追求：

- 海报化、宣传片化视觉
- 每页都做重装饰
- 把所有模块都详细展开
- 在资格赛匿名版中放入成员介绍
- 把 PPT 变成系统操作手册

## 实施建议

在用户确认本设计说明后，实施阶段应按以下顺序推进：

1. 先生成匿名资格赛版 PPT
2. 完成视觉 QA 与内容 QA
3. 再基于匿名版衍生一份非匿名展示版
4. 将资格赛版输出到用户指定目录，供后续替换真实截图
